# 七牛断点续传改造说明

这份文档是新增说明文档。

目的：

- 说明在现有“标准表单上传”基础上，如果要支持七牛断点续传，需要调整哪些内容
- 说明七牛侧是否需要额外设置
- 说明服务端和客户端分别需要做哪些改动

适用前提：

- 当前方案是“服务端下发上传配置，客户端按表单直传到七牛”
- 希望后续支持“大文件上传”和“断点续传”

---

## 1. 先说结论

### 结论 1：断点续传不是在现有表单上传上加一个参数就能完成

你们当前标准表单上传，本质上是：

- 一个 `POST`
- 一次性把整个文件传上去

这种方式适合：

- 小文件
- 简单接入
- 低改造成本

但它不适合真正的断点续传。

七牛官方文档明确区分了：

- 表单上传
- 分片上传

并说明断点续传是建立在分片上传基础上的。  
官方也明确写到：表单上传适合较小文件；分片上传适合较大文件，并支持断点续传。  
参考：
- [上传资源](https://developer.qiniu.com/kodo/manual/1234/upload-types)
- [上传文件](https://developer.qiniu.com/kodo/8562/upload-a-file)

### 结论 2：如果要支持断点续传，客户端上传模式必须切换

也就是说：

- 现有 `form_post`
- 不能直接升级成“真正的断点续传”

要支持断点续传，客户端需要改成以下两种之一：

1. 使用七牛官方 SDK 的分片上传/断点续传能力
2. 自己实现七牛分片上传 API（更复杂，不推荐作为第一选择）

### 结论 3：七牛控制台一般不需要额外开一个“断点续传开关”

通常不需要在七牛后台额外开启一个特殊功能开关。  
重点不在控制台开关，而在于：

- 上传方式要从表单上传改成分片上传
- 上传凭证要仍然合法
- 上传域名要正确
- 客户端要保存分片进度

换句话说：

- 断点续传更多是“接入方式变化”
- 不是“后台点一个配置就行”

## 2. 七牛官方能力说明

七牛官方文档里的关键点如下：

1. 表单上传适合小文件，不适合断点续传  
来源：
- [上传资源](https://developer.qiniu.com/kodo/manual/1234/upload-types)

2. 分片上传适合大文件，并支持断点续传  
来源：
- [上传资源](https://developer.qiniu.com/kodo/manual/1234/upload-types)
- [上传文件](https://developer.qiniu.com/kodo/8562/upload-a-file)

3. 分片上传 v2 是推荐方案  
来源：
- [分片上传 v2 版](https://developer.qiniu.com/kodo/manual/7458/shard-upload-v2-version)
- [分片上传 v2 API](https://developer.qiniu.com/kodo/api/6364/multipartupload-interface)

4. 断点续传本质上要保存本地上传进度，再继续未完成分片  
来源：
- [Go SDK 文档](https://developer.qiniu.com/kodo/1238/go?category=sdk)
- [JavaScript SDK 文档](https://developer.qiniu.com/kodo/sdk/1283/javascript)

## 3. 七牛侧需要特别设置什么

### 一般不需要额外开关

通常不需要专门在七牛控制台打开“断点续传开关”。

### 需要确认的内容

虽然通常不用额外开功能，但下面这些需要确认：

1. `bucket` 正常可用
2. 上传域名正确
3. 上传凭证有效
4. 上传文件大小、类型限制是否合理
5. 如果走 SDK，要确认客户端平台支持对应 SDK

### 需要注意的点

1. 如果你们当前下发的是“纯表单上传配置”，那这份配置本身不够
2. 断点续传时，客户端会多次请求七牛分片接口，不再是一次上传完成
3. 如果 token 有效期太短，上传到一半可能过期，需要考虑续期策略

---

## 4. 服务端需要调整什么

服务端改动分成“必须调整”和“建议调整”两类。

### 4.1 必须调整

#### 1. 上传协议要从“表单直传”扩展成“支持分片上传”

当前你们下发的大概是这种结构：

```json
{
  "provider": "qiniu",
  "mode": "form_post",
  "uploadUrl": "https://xxx",
  "uploadToken": "xxxxx",
  "keyPrefix": "DeviceName/Date/"
}
```

如果要支持断点续传，服务端至少要让客户端知道：

- 当前上传模式不是普通表单上传
- 当前文件应该走分片上传

所以至少要把协议扩展成类似：

```json
{
  "provider": "qiniu",
  "mode": "resumable",
  "uploadToken": "xxxxx",
  "keyPrefix": "DeviceName/Date/",
  "partUploadVersion": "v2"
}
```

说明：

- 字段名只是示意
- 核心是要告诉客户端“这是断点续传模式”

#### 2. token 有效期要重新评估

现在表单上传一般时间较短也够用。  
但断点续传下：

- 文件可能很大
- 上传可能跨更长时间
- 可能因为网络中断而延迟完成

所以需要确认：

- `expiresInSec` 是否足够
- token 过期后是否允许客户端重新获取

#### 3. 服务端结果回传结构要兼容分片上传状态

以前回传的是“单次上传成功/失败”。

断点续传后，建议服务端容纳更多信息：

- 是否开始分片上传
- 是否恢复了历史进度
- 最终是否完成文件合成
- 错误发生在哪个阶段

虽然最终仍然可以只落单个文件结果，但服务端至少要接受更长链路的状态变化。

### 4.2 建议调整

#### 1. 下发更明确的文件大小阈值

可以由服务端下发一个阈值，比如：

- 小于某值：仍走表单上传
- 大于某值：走断点续传

这样客户端可同时支持两种模式。

#### 2. 下发上传模式选择规则

例如：

```json
{
  "uploadStrategy": {
    "smallFileMode": "form_post",
    "largeFileMode": "resumable",
    "largeFileThreshold": 4194304
  }
}
```

这里的 `4194304` 是 4MB，只是示例。  
七牛官方文档提到，单文件大于 4MB 时，建议使用分片上传和断点续传。  
来源：
- [上传文件](https://developer.qiniu.com/kodo/8562/upload-a-file)

#### 3. 设计 token 续签方案

大文件断点续传下，建议预先定义：

- token 过期时客户端如何处理
- 是否重新向服务端申请新 token
- 断点信息是否继续复用

---

## 5. 客户端需要调整什么

客户端改动是这次的重点。

### 5.1 必须调整

#### 1. 不能再只支持“表单一次性上传”

客户端目前如果只会做：

- `token + key + file`
- 一次 `multipart/form-data`

那么它不具备断点续传能力。

客户端必须新增一套“分片上传执行逻辑”。

#### 2. 需要支持分片上传流程

断点续传不是只上传一个文件，而是：

1. 初始化分片上传任务
2. 按 Part 上传分片
3. 保存已上传分片进度
4. 中断后恢复
5. 最后完成文件合成

如果走七牛分片上传 v2，自实现时至少要处理：

- 初始化
- 上传分片
- 完成上传
- 必要时终止上传

参考：
- [分片上传 v2 版](https://developer.qiniu.com/kodo/manual/7458/shard-upload-v2-version)
- [分片上传 v2 API](https://developer.qiniu.com/kodo/api/6364/multipartupload-interface)

#### 3. 需要在本地持久化上传进度

“断点续传”的关键不只是分片，而是：

- 上传到哪一片了
- 哪些片已经成功
- 下次能从哪里恢复

所以客户端要新增本地持久化能力，例如保存：

- 文件唯一标识
- 文件大小
- key
- uploadId
- 已成功的 part 列表
- 每个 part 的 etag
- 进度文件更新时间

#### 4. 需要支持失败恢复

断点续传要处理这些场景：

- App 被杀
- 设备重启
- 网络断开
- token 过期
- 部分分片成功、部分失败

客户端逻辑要能恢复，而不是重新全量上传。

### 5.2 建议调整

#### 1. 同时保留表单上传能力

不建议把所有文件都强制切成断点续传。  
更合理的是：

- 小文件继续表单上传
- 大文件走断点续传

#### 2. 增强上报信息

客户端回传服务端时，建议补充：

- 上传模式：`form_post` / `resumable`
- 是否断点恢复：`is_resumed`
- 已上传大小
- 分片失败原因
- 最终文件合成是否成功

---

## 6. 推荐的改造方式

### 方案一：客户端接七牛官方 SDK

这是更推荐的第一选择。

优点：

- 七牛官方已经封装好大量上传细节
- 断点续传逻辑更成熟
- 少自己踩分片协议细节坑

缺点：

- 客户端需要引入七牛 SDK
- 会增加一定平台适配工作

### 方案二：客户端自己实现七牛分片上传 v2 API

优点：

- 控制力更强
- 不依赖 SDK

缺点：

- 实现复杂度高
- 容易在分片状态、重试、恢复、合并上踩坑

所以一般建议：

- 优先评估官方 SDK
- 除非有非常明确的原因，否则不建议第一步就自研分片协议

---

## 7. 对现有文档体系的影响

如果你们决定支持断点续传，现有文档体系至少要补充这些说明：

1. 上传模式增加 `resumable`
2. 服务端下发配置增加断点续传相关字段
3. 手机端/daemon 新增本地进度保存说明
4. 上传结果回传增加断点续传状态信息

也就是说：

- 不是推翻现有文档
- 而是在现有“表单上传”之外，再增加一套“断点续传上传模式”

---

## 8. 最终建议

如果只是希望快速支持七牛断点续传，建议按下面顺序推进：

1. 先保留现有表单上传不变
2. 增加“大文件走断点续传”的新模式
3. 服务端下发明确的上传模式和阈值
4. 客户端优先接七牛官方 SDK
5. 回传链路补充分片/恢复相关状态

---

## 9. 一句话总结

“七牛断点续传不是后台开关问题，而是上传模式从表单直传切换为分片上传的问题；七牛侧通常不需要额外开功能，但服务端要扩展上传协议，客户端要新增分片、进度持久化和恢复能力。”
